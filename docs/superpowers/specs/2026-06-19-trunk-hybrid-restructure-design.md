# Track 2 — Trunk-hybrid orchestration restructure (design + binding must-fixes)

> Founder directive: collapse the orchestrator + `know-things` into ONE **trunk** agent that handles general convo + USC knowledge (`know-things` becomes a skill: its tools available on the trunk, its domain rules loaded via the skills system). KEEP exactly two dispatched sub-agents: `find-people` (squad) and `what's-happening` (events). Behind `GEORGE_TRUNK_HYBRID` (default-OFF, byte-for-byte rollback to today's 3-sub-agent dispatch). Produced by the design workflow (4 mappers → design → adversarial critic).

## Target architecture

END STATE (behind GEORGE_TRUNK_HYBRID=true; default OFF → today's 3-sub-agent dispatch byte-for-byte):

TRUNK = one agent running buildTrunkPrompt() = MASTER_PROMPT + ORCHESTRATOR_PROMPT + KNOW_THINGS_PROMPT (inlined as domain guidance, NOT the other two domains) + BATCH/COURSE-FASTPATH guidance + the same overlay stack the orchestrator already builds (mood, activity, delay, world-state, profile, relationship note, onboarding nudge, studentId) + the skill catalog. The trunk holds, directly in its allowedTools: every know-things tool (campus_knowledge, freshman_faq, describe_course, recommend_courses, get_rmp_ratings, search_courses, search_ge_courses, ge_candidates, search_programs, plan_schedule, get_student_academic_state, course_tips, get_course_reviews, search_roommates, search_sublets, post_sublet, find_places, dps_zone_check) PLUS the 2 ex-orchestrator direct tools (set_reminder, load_skill) PLUS the dispatch tools (Task, Agent) PLUS WebSearch when under cap. It does NOT hold find-people or what's-happening tools.

DISPATCH: exactly TWO sub-agents remain in queryOptions.agents — find-people and whats-happening — each unchanged (same prompt, same tool list, same model tier, same per-turn overlay injection via buildAgentsConfig). The trunk's prompt instructs: answer course/housing/immigration/campus directly with your own tools; dispatch to find-people only for squad organizing/joining and to whats-happening only for events discovery. Routing collapses from 'pick 1 of 3' to 'answer directly, or dispatch 0-or-1 only for social/event'. The single most critical invariant (find-people's approval gate, privacy red lines, category validation, ping fan-out side effects) is preserved precisely because find-people stays a dispatched sub-agent with its own specialization prompt — nothing about it merges into the trunk.

This is the SINGLE_AGENT seam evolved: instead of one all-domains agent (3 specializations inline, all tools), the trunk inlines ONLY know-things and keeps find-people + whats-happening as dispatch targets. Model tier: trunk runs SMART_MODEL (it now owns high-stakes know-things reasoning), eliminating the prior FAST-orchestrator→SMART-know-things hop. Telemetry, MCP registration, namespacing, fast-path, settingSources:[], admin gate, world-state observe/render, studentId resolution, WebSearch budgeting all unchanged.

## know-things → skill (mechanics)

Two complementary moves, both reusing existing seams; nothing is lost:

(A) THE 14 (actually 18) TOOLS BECOME DIRECTLY AVAILABLE TO THE TRUNK — via the tool SET, not via the skills loader. They are already all registered in ALL_TOOLS (src/tools/index.ts) and already in the georgeToolServer MCP (createSdkMcpServer with alwaysLoad:true). So 'directly available' = adding their namespaced names (nsTool) to the trunk's allowedTools. I do this by adding a single exported const TRUNK_TOOLS in agents.config.ts = [...SUB_AGENTS['know-things'].tools, ...ORCHESTRATOR_DIRECT_TOOLS] (set_reminder, load_skill move from orchestrator-direct onto the trunk). The trunk's allowedTools = ['Task','Agent', ...TRUNK_TOOLS.map(nsTool), ...(webAllowed?['WebSearch']:[])]. No tool plumbing/registration changes — every tool object already exists and is already in the MCP server.

(B) ITS DOMAIN RULES LOAD VIA THE SKILLS SYSTEM (catalog + load_skill) AND the unified trunk prompt — the existing two channels, no new machinery:
  1. The bulk of know-things domain rules already live in prompts/know-things.md (KNOW_THINGS_PROMPT). buildTrunkPrompt inlines that string verbatim after ORCHESTRATOR_PROMPT. This is the same inlining technique buildSingleAgentPrompt already uses for UNIFIED_DOMAIN_PROMPT — just narrowed to one domain. Anti-fabrication/voice stays in MASTER_PROMPT (unchanged) so voiceLint + master.md guardrails are universal.
  2. Reusable playbooks (workflow steps, tool-call order, fallback logic) become SKILL files under src/skills/sub-agents/ using the EXISTING SubAgent enum values course | housing | campus (already in types.ts — NO enum change needed, NO loader/registry test churn). Concretely seed: course/course-recommendation.md (tools: recommend_courses, ge_candidates, get_rmp_ratings — encodes 'call recommend_courses/ge_candidates first', RMP thresholds, WRIT-150 5.0, • bullet formatting), course/ge-category-picker.md (search_ge_courses, get_rmp_ratings), housing/find-housing.md (search_sublets, search_roommates, post_sublet, dps_zone_check — dorm picks, safety-circle, never-invent-prices), campus/campus-knowledge-fallback.md (campus_knowledge, freshman_faq, find_places — meal-plan/dining/study-spot specifics, fallback strategy). Each validates at buildRegistry because every referenced tool is in ALL_TOOLS. These are ADDITIVE (new files) — they appear in getFullCatalog() automatically, harmless when the flag is off.
  3. The trunk DISCOVERS these via getFullCatalog() (already emits orchestrator + every sub-agent bucket incl. course/housing/campus) appended to buildTrunkPrompt exactly as buildSingleAgentPrompt already appends it. Trunk pattern-matches a description and calls load_skill({name}); load_skill (unchanged) returns the body. Profile-based personalization (major/year/interests for recs) is preserved because buildTrunkPrompt injects buildUserProfileBlock just like the orchestrator/single paths.

agents.config.ts changes: add TRUNK_TOOLS const; keep SUB_AGENTS['know-things'] defined (so rollback path + UNIFIED_DOMAIN_PROMPT stay intact) but it is simply NOT included in the trunk-path agents map. prompts changes: new buildTrunkPrompt in orchestrator.ts (clone of buildOrchestratorPrompt + KNOW_THINGS_PROMPT/web/catalog); a flag-gated branch in prompts/orchestrator.md is NOT needed — instead append a short trunk-routing addendum string in buildTrunkPrompt ('handle USC knowledge yourself; dispatch only for squad/events') so prompts/orchestrator.md stays byte-for-byte unchanged for the OFF path.

## Flag plan

Single new flag GEORGE_TRUNK_HYBRID (default OFF, read as `=== 'true'` exactly like SINGLE_AGENT). Three-way precedence in orchestrator.ts: if GEORGE_TRUNK_HYBRID → trunk+2-subagent path; else if SINGLE_AGENT → existing single-agent path (untouched); else → existing multi-agent 3-sub-agent dispatch (untouched, the default). The flag is documented in .env.example beside SINGLE_AGENT. The two existing branches are moved verbatim into the else-if/else, so when the flag is off the produced queryOptions are byte-for-byte what they are today. New skill files are additive and only surface via getFullCatalog(), which only the trunk (or pre-existing single-agent) path injects — so they are inert when the flag is off.

## Rollback

Set GEORGE_TRUNK_HYBRID=false (or unset) → orchestrator falls through to the unchanged multi-agent ternary and dispatches to all 3 sub-agents exactly as today; no redeploy of prompts/tools needed because SUB_AGENTS['know-things'], UNIFIED_DOMAIN_PROMPT, ORCHESTRATOR_DIRECT_TOOLS, and prompts/orchestrator.md are all left intact. The 4 new skill files remain on disk but are never injected on the OFF path, so they have no effect. To fully revert the code: delete buildTrunkPrompt/buildTrunkAgentsConfig/TRUNK_TOOLS/TRUNK_MODEL and the trunkHybrid branch; the diff is isolated and additive, so reverting cannot disturb the live default path. Burn-in discipline mirrors the Spectrum migration: keep the 3-agent path as the rollback target through a burn-in window before considering deletion.

## Design-noted risks

- Approval-gate erosion if know-things ever bleeds into find-people: mitigated by NOT inlining find-people into the trunk — find-people stays a dispatched sub-agent so its prompt-enforced approval gate, privacy red lines, category (约会) rejection, and ping fan-out reasoning are untouched. The trunk holds zero squad tools, so it physically cannot call create_squad_post.
- Model-cost shift: trunk runs SMART_MODEL for every non-fast-path turn (vs today's FAST orchestrator). Small-talk is still caught by the unchanged fast-path, but borderline factual turns now pay SMART once instead of FAST-orchestrator+SMART-know-things. Net likely cheaper (one fewer hop) but watch per-turn cost telemetry during burn-in; resolveModelForUser still allows per-user override.
- Dispatch under-triggering: with know-things inlined, the trunk might try to answer a squad/event request directly. Mitigated by TRUNK_ROUTING_ADDENDUM explicitly scoping dispatch, and by the trunk lacking squad/event tools (it cannot fulfill those without dispatching). Verify in E2E.
- WebSearch allocation: today webSearchGuidance is injected for whats-happening+know-things sub-agents; in trunk mode know-things web guidance must move to buildTrunkPrompt (done in step 3) while whats-happening keeps its own via buildAgentsConfig. Risk of double-injection if buildTrunkAgentsConfig still web-blocks know-things — but know-things is removed from the trunk agents map, so no double-count.
- Skill-catalog noise: getFullCatalog lists every sub-agent bucket incl. event/social skills the trunk should dispatch rather than load. Mitigated because load_skill only returns a playbook body (no execution) and the routing addendum tells the trunk to dispatch social/event; worst case the trunk loads a playbook it should have dispatched — degraded, not unsafe. A later refinement could give the trunk a know-things-only catalog via getCatalogFor, but that is out of scope for the byte-safe first cut.
- Telemetry attribution change: direct know-things turns now show tools but no subAgent (previously showed subAgent='know-things'). Dashboards keying on subAgent='know-things' will see a drop. Acceptable/expected, but flag it to Bobby since web chat hits real users and dashboards are watched.
